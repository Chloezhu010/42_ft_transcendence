# Tech Debt

Known design issues that are working as-is but should be revisited. Each entry records the symptom, the root cause, the current workaround, and the proper fix.

## `POST /api/stories` round-trips all images through JSON

### Symptom

Saving a freshly generated story from the preview/edit flow failed with `Failed to save story: request entity too large`. The browser received a `413` from nginx before the request reached FastAPI.

### Root cause

The "generate → preview → edit → save" flow round-trips every image through the client as base64 inside a single JSON body:

1. Frontend calls `POST /api/generation/panel-image` per panel. The backend generates the image via Gemini and returns it inline as base64 (`backend/routers/generation.py:146`) — nothing is written to disk.
2. Frontend holds those base64 strings in memory while the user previews and optionally edits panels.
3. On save, frontend sends `POST /api/stories` with `cover_image_base64` plus `image_base64` for every panel. The backend only now writes the bytes to `backend/images/` via `save_base64_image` (`backend/db/crud.py:29`).

Story scripts currently request 10 panels (`backend/llm/gemini_service.py:107`), so one save request carries **11 base64-encoded PNGs** (cover + 10 panels). Gemini's 1024×1024 PNGs are typically 1–2 MB raw, ~1.3–2.7 MB after base64. Worst case the body exceeds 20 MB, which trips nginx's default `client_max_body_size` of 1 MB.

Note: the alternative path `POST /api/stories/generate` (`backend/routers/generation.py:38`) does generation and persistence in one server-side call and never hits this ceiling. Only the user-editable preview flow is affected.

### Current workaround

`nginx/nginx.conf` sets `client_max_body_size 25m` on the 443 server block. This is just large enough to cover the current worst case with a small safety margin. It is **not** a principled limit — it is coupled to the number of panels and image resolution, so any change to those values will require revisiting it.

### Proper fix

Stop shipping image bytes through the save endpoint. The backend already had the image in memory when it generated it; there is no reason to ask the client to hand the same bytes back.

Preferred approach:

1. Change `POST /api/generation/panel-image` and `POST /api/generation/edit-panel-image` to persist the generated image to `backend/images/` immediately and return `{id, url}` instead of `image_base64`.
2. Change `StoryCreate` / `StoryUpdatePanels` to accept `image_id` per panel instead of `image_base64`. The backend links the existing file to the story on save.
3. Add a cleanup job for orphan images (generated but never attached to a saved story) — e.g. delete files older than N hours with no panel reference.

After the fix:

- `POST /api/stories` body shrinks from tens of MB to a few KB.
- `client_max_body_size` can be lowered to 2–4 MB (still large enough for `KidProfileCreate.photo_base64` on profile creation) and is no longer coupled to panel count or image resolution.
- Network round-trips for generated bytes are halved.
